# Velkor Consultive Platform — Architecture Reference

> Operational intake intelligence layer built on Next.js 14.  
> Transforms the marketing site into a lead qualification and operational risk assessment pipeline.

---

## Overview

The platform implements a multi-stage consultive intake flow:

```
Visitor → 5-Step Wizard → API Route → Scoring Engine → Lead Classification
                                                       ↓
                               Strapi CRM ← Storage   PDF Report (client)
                               Webhook Pipeline ←       ↓
                               n8n / Make / Zapier    Results Screen
```

## 1. Scoring Engine (`lib/scoring.ts`)

### Model

Composite maturity score across three domains, weighted to reflect Velkor's service expertise and Mexican market risk profile:

| Domain | Weight | Rationale |
|--------|--------|-----------|
| Identity & Governance | **40%** | Highest-impact control in M365-heavy SMB/mid-market |
| Infrastructure | **35%** | Foundational — segmentation, perimeter, continuity |
| Operations | **25%** | Process maturity indicator, not direct attack surface |

### Domain Scoring

**Infrastructure** (max 100 pts)

| Control | Points | Rationale |
|---------|--------|-----------|
| VLAN segmentation | 20 | Lateral movement prevention |
| Firewall quality (NGFW) | 20 | Perimeter enforcement quality |
| Backup strategy (cloud-verified) | 20 | Business continuity critical |
| Endpoint inventory | 15 | Visibility prerequisite |
| VPN (business-grade) | 10 | Remote access control |
| WiFi segmentation | 8 | Network hygiene |
| EDR/AV | 7 | Endpoint threat detection |

**Identity** (max 100 pts)

| Control | Points | Rationale |
|---------|--------|-----------|
| MFA enforcement (all users) | 30 | Single highest-impact identity control |
| Conditional Access (configured) | 20 | Policy-based access enforcement |
| Entra ID (full) | 15 | Identity foundation |
| Privileged access (PAM) | 15 | Admin account security |
| Intune MDM (full) | 15 | Device posture enforcement |
| Onboarding/offboarding (automated) | 5 | Lifecycle governance |

**Operations** (max 100 pts)

| Control | Points | Rationale |
|---------|--------|-----------|
| Monitoring (SIEM) | 25 | Detection capability |
| Last security audit (recent) | 20 | Assessment discipline |
| IT documentation (current) | 15 | Operational knowledge foundation |
| Change management (formal) | 15 | Control environment |
| Compliance framework active | 15 | Governance maturity signal |
| Ticketing system (formal) | 10 | Incident tracking |

### Maturity Levels

| Range | Level | Label |
|-------|-------|-------|
| 0–24 | `initial` | Inicial |
| 25–44 | `developing` | Desarrollo |
| 45–64 | `defined` | Definida |
| 65–79 | `managed` | Gestionada |
| 80–100 | `optimized` | Optimizada |

### Exposure Levels

Derived from flag count, not raw score:

| Condition | Exposure |
|-----------|----------|
| Any critical flag | `critical` |
| Any high flag (no critical) | `high` |
| Only medium/low flags | `medium` |
| Zero flags | `managed` |

### Risk Flag Logic

Flags are generated by `buildFlags()` based on specific answer patterns:

**Critical flags triggered by:**
- MFA = `none`
- Privileged access = `shared` accounts
- Backup = `none`
- Urgency = `critical` (active incident declared)

**High flags triggered by:**
- VLAN segmentation = `no` (flat network)
- Endpoint inventory = `none` OR Intune = `none`
- Firewall = `isp-default` or `none`
- Monitoring = `none`
- Conditional Access = `none` while Entra ID is configured

**Medium flags triggered by:**
- Backup = `local` only (no offsite)
- Documentation = `none`
- Last audit = `never`
- WiFi segmentation = `no`
- Onboarding process = `none`

## 2. Lead Classification Engine (`lib/classification.ts`)

### Segments

| Company Size | Segment |
|-------------|---------|
| 1–25, 26–100 | SMB |
| 101–500 | Mid-Market |
| 500+ | Enterprise |

### Engagement Types (priority order)

1. **Compliance-Driven** — active compliance frameworks or compliance needs declared
2. **Security Remediation** — urgency = `critical` (active incident)
3. **Identity Governance** — 2+ identity/governance flags AND identity score < 40
4. **Infrastructure Refresh** — 2+ infra/endpoint flags AND infra score < 40
5. **Modernization** — partial M365 or unformalized infra + modernization goal
6. **Managed Services** — overall score ≥ 60 (mature org needing continuity)
7. **Comprehensive Audit** — default for complex or unclear profiles

### Priority & Follow-up SLA

| Priority | Trigger | Channel | SLA |
|----------|---------|---------|-----|
| `critical` | Any critical flag OR urgency = critical | Phone call | 4 h |
| `high` | 2+ high flags OR urgency = urgent OR score < 45 | Phone call | 8 h |
| `normal` | Standard | Technical email + scheduling | 24 h |
| `low` | urgency = evaluating | Async proposal | 48 h |

Enterprise segment always escalates to phone-call regardless of priority.

## 3. Assessment API (`app/api/assessment/route.ts`)

### Request flow

```
POST /api/assessment
  1. IP rate limit check (3 req / 60 s / IP) — silent 200 on exceed
  2. JSON parse
  3. Honeypot check (_hp field) — silent 200 on trigger
  4. Turnstile verification (opt-in — pass-through if TURNSTILE_SECRET_KEY unset)
  5. Contact validation + sanitization
  6. Disposable email check — 400 with message
  7. Step field extraction + sanitization
  8. scoreAssessment() — CIS Controls v8 weighted scoring
  9. classifyLead() — engagement type, priority, SLA
  10. reportRef = VLK-{base36 timestamp suffix}
  11. Strapi /leads POST (assessment summary embedded in notes field)
  12. dispatchWebhook('assessment.completed', ...)
  13. Conditional: dispatchWebhook('lead.critical', ...) if criticalCount > 0
  14. Conditional: dispatchWebhook('lead.enterprise', ...) if Enterprise or critical priority
  15. logAuditEvent('submission', ...)
  16. Return { ok: true, reportRef, scores, classification }
```

## 4. Automation Pipeline (`lib/webhook.ts`)

### Configuration

| Env Var | Required | Purpose |
|---------|----------|---------|
| `ASSESSMENT_WEBHOOK_URL` | No | Target URL (n8n, Make, Zapier) — disabled if unset |
| `ASSESSMENT_WEBHOOK_SECRET` | No | HMAC-SHA256 signing key — unsigned if unset |

### Event catalog

| Event | Trigger | Payload key fields |
|-------|---------|-------------------|
| `assessment.completed` | Every submission | company, email, segment, engagementType, priority, overallScore, criticalFlags, maturity, exposure, urgency |
| `lead.critical` | criticalCount > 0 OR priority = critical | company, email, reportRef, flags[] |
| `lead.enterprise` | segment = Enterprise OR priority = critical | company, email, segment, reportRef |
| `assessment.pdf_generated` | *(reserved — not yet fired)* | — |
| `lead.scored` | *(reserved)* | — |
| `followup.scheduled` | *(reserved)* | — |

### Payload envelope

```json
{
  "event": "assessment.completed",
  "timestamp": "2026-05-13T19:47:00.000Z",
  "source": "velkor-web",
  "version": "1",
  "data": { ... }
}
```

### HMAC verification (receiver side)

```js
// Verify X-Velkor-Signature header
const hmac = createHmac('sha256', WEBHOOK_SECRET)
hmac.update(rawBody)
const expected = hmac.digest('hex')
const received = req.headers['x-velkor-signature']
// Use timingSafeEqual to prevent timing attacks
```

## 5. PDF Report Architecture (`lib/reportHtml.ts`)

### Approach

Zero-dependency server-side generation using print-optimized HTML.  
No `jspdf`, `@react-pdf/renderer`, `puppeteer`, or browser automation.

### Flow

```
ResultsScreen (client) → generateReportHtml(data) → HTML string
                       → window.open('', '_blank') → new tab
                       → win.document.write(html)
                       → win.document.close()
                       → User clicks "Imprimir / Guardar PDF" → window.print()
                       → Browser PDF dialog → Save as PDF
```

### Report sections

1. Header (company, industry, size, date, reportRef)
2. Maturity score + exposure badge
3. Domain score bars (infrastructure / identity / operations)
4. Risk flags (up to 8, severity-colored with left border)
5. Immediate priorities (numbered list)
6. Quick wins (30–90 day actions)
7. Operational observations
8. Recommended phase + estimated scope
9. Context summary (6-field grid)
10. Footer (Velkor System, velkor.mx, report date, ref)

### Print CSS strategy

- Dark theme: `#08090f` background (screen display)
- Print override: `background: #ffffff; color: #1a1a2a`
- `@page { size: A4; margin: 20mm 16mm; }`
- `.no-print { display: none !important; }` hides the print button on paper
- Score bars: `width: ${score}%` inline style (no JS required for print)

## 6. Security Hardening

### Layers (defense in depth)

1. **Rate limiting** — in-memory Map, 3 req / IP / 60 s, silent 200 passthrough
2. **Honeypot** — off-screen `<input name="_hp">` (`position:absolute;left:-9999px` — NOT `display:none`, which bots skip)
3. **Turnstile** — Cloudflare CAPTCHA, opt-in via `TURNSTILE_SECRET_KEY`, fail-open on network error
4. **Disposable email** — ~50-domain blocklist, 400 with message to user
5. **Input sanitization** — `san(v, maxLen)` strips HTML tags + trims + slices; `sanArr` for arrays
6. **Audit logging** — structured JSON to stdout; Railway log aggregation captures automatically

### Env vars required for hardening

| Var | Effect if set |
|-----|---------------|
| `TURNSTILE_SECRET_KEY` | Enables Cloudflare Turnstile bot protection |
| `NEXT_PUBLIC_TURNSTILE_SITE_KEY` | Enables frontend widget rendering *(not yet wired)* |
